# Task: SBC Optimization Review

## Objective

Perform a comprehensive codebase review focused on optimization for SBC (Single Board Computer) and low-resource environments. Generate a detailed report with findings, recommendations, and prioritized next steps.

---

## Context

Weaver Desktop targets resource-constrained devices:

- Raspberry Pi Zero/1/2/3/4/5
- 7" touchscreen displays
- Limited RAM (256MB - 512MB)
- Limited CPU
- Battery-powered cyberdecks

**Key Principles:**

- Avoid heap allocations where possible
- Prefer static function pointers over boxed closures
- Use stack-allocated buffers (ArrayString, arrays)
- Cache computed values to reduce per-frame work
- Minimize syscalls in hot paths

---

## Subtasks

### 1. Heap Allocation Audit

Scan for heap-allocating patterns:

- [ ] `Box<dyn ...>` - boxed trait objects
- [ ] `Vec<T>` - where fixed-size arrays would work
- [ ] `String` - where `&'static str` or `Cow<'static, str>` would work
- [ ] `HashMap`/`HashSet` - consider alternatives for small collections
- [ ] `.clone()` calls on heap types
- [ ] `format!()` macro usage in hot paths
- [ ] `.to_string()` / `.to_owned()` calls

**Report:** List each occurrence with file, line, and recommended fix.

---

### 2. Per-Frame Allocation Check

Identify allocations that happen every frame (inside `ui()` or `update()` methods):

- [ ] String formatting in render loops
- [ ] Creating new collections each frame
- [ ] Cloning data unnecessarily
- [ ] Repeated syscalls (e.g., `chrono::Local::now()` every frame)

**Report:** Flag each with severity (HIGH/MEDIUM/LOW) and mitigation.

---

### 3. Reactive System Validation

Verify the reactive primitives follow zero-allocation patterns:

- [ ] `SignalFn<T>` uses `fn(&T)` pointers only
- [ ] `Observable<T>` uses `SignalFn` (not boxed closures)
- [ ] `Signal<T>` uses `SignalFn` (not boxed closures)
- [ ] `Interactable<T>` uses `SignalFn` for all events
- [ ] No `Box<dyn Fn>` in event handling

**Report:** Confirm compliance or list violations.

---

### 4. Static Data Opportunities

Find data that could be compile-time static:

- [ ] Button labels that never change → `&'static str`
- [ ] Menu items defined at startup → fixed arrays
- [ ] Configuration defaults → `const` values
- [ ] Color constants → `const Color32`

**Report:** List opportunities with estimated impact.

---

### 5. Memory Layout Review

Check struct layouts for efficiency:

- [ ] Large structs that could use `Box` for rare data
- [ ] Enum variants with vastly different sizes
- [ ] Padding waste in struct field ordering
- [ ] Consider `#[repr(C)]` where beneficial

**Report:** Note any structs worth optimizing.

---

### 6. Unused Code Cleanup

Identify dead code:

- [ ] Unused imports
- [ ] Unused functions/methods
- [ ] Commented-out code blocks (evaluate if needed)
- [ ] Old files not referenced in `mod.rs`

**Report:** List for cleanup.

---

### 7. Best Practices Check

Validate Rust idioms for performance:

- [ ] `#[inline]` on hot path functions
- [ ] `const fn` where possible
- [ ] Prefer iterators over index loops
- [ ] Use `&str` over `&String` in function signatures
- [ ] Avoid `unwrap()` in production paths (use `expect()` with context)

**Report:** List improvements.

---

### 8. External Dependencies Review

Evaluate dependency impact:

- [ ] Check for heavy dependencies that could be replaced
- [ ] Verify feature flags minimize bloat
- [ ] Consider `no_std` compatible alternatives where applicable

**Report:** List any concerning dependencies.
